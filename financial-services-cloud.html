    Financial Services Cloud

    O Financial Services Cloud (FSC) expande o CRM padrão da Salesforce com modelos de dados e fluxos específicos para bancos, seguradoras e gestoras de investimento. As seções abaixo oferecem o conteúdo completo do plano de estudo por nível.

    Nível Iniciante

    Explicação: FSC estende padrões de CRM para modelar relacionamentos financeiros, agrupamentos familiares e ativos de clientes.

    Caso de uso: Um banco deseja rastrear contas correntes, investimentos e metas de aposentadoria de seus clientes.

    Tópicos teóricos:

    • Household e PersonAccount: agrupamento de membros de uma família versus contas individuais.
    • FinancialAccount e FinancialGoal: tipos (Banking, Investment, Insurance) e ciclos de vida.
    • Guided Engagement & Path: configuração de etapas de atendimento e guidance aos advisors.

    Hands‑on Lab:

    • Criar um Household com 2 PersonAccounts e associar 3 FinancialAccounts (Bank e Investment).
    • Definir um FinancialGoal "Retirement" e configurar Path Guidance para orientar o advisor.

    Testes de Conhecimento: Quando usar Household em vez de Account padrão? Qual tipo de FinancialAccount se aplica a um seguro de vida?

    Exercícios Práticos:

    • Personalizar a Lightning Record Page para o advisor mostrando resumo financeiro.
    • Configurar uma Permission Set para restringir acesso a dados sensíveis de contas.

    Nível Intermediário

    Explicação: Compliance e relatórios garantem segurança de dados e oferecem insights sobre patrimônio e performance.

    Caso de uso: Uma instituição financeira deve criptografar dados sensíveis e gerar relatórios consolidados sobre performance de advisors e patrimônio dos clientes.

    Tópicos teóricos:

    • Field‑Level Encryption & Masking: uso do Shield para encriptar e mascarar campos sensíveis.
    • Relatórios e dashboards multi‑objeto para consolidar dados de Household, FinancialAccount e Advisor.
    • Ferramentas de produtividade de advisors: workspaces, action plans e checklists de atendimento.

    Hands‑on Lab:

    • Aplicar Field‑Level Encryption ao campo SSN__c e validar a máscara em diferentes perfis.
    • Criar um Joined Report combinando Household e FinancialAccount com métricas de patrimônio.
    • Construir um dashboard de performance de advisors com componentes gráficos.

    Testes de Conhecimento: Qual a diferença entre Shield Encryption e Classic Encryption? Como criar um relatório Joined entre objetos customizados e padrão?

    Exercícios Práticos:

    • Configurar um Action Plan Template para processos de onboarding de clientes.
    • Agendar uma exportação de relatório diário via Data Export Service.

    Nível Avançado

    Explicação: Componentes LWC personalizados e integrações bancárias ampliam as capacidades do FSC.

    Caso de uso: Exibir um gráfico de evolução patrimonial em tempo real utilizando dados de uma fintech externa.

    Tópicos teóricos:

    • LWC & D3.js: criação de componentes Lightning para visualização avançada de dados.
    • Batch Apex: conciliação diária de transações financeiras.
    • Streaming API e Platform Events: captura de eventos de transação em tempo real.

    Hands‑on Lab:

    • Desenvolver um LWC que consome uma API REST externa e exibe um gráfico D3.js dos ativos financeiros.
    • Escrever um Batch Apex para processar transações pendentes e atualizar o saldo de FinancialAccounts.
    • Configurar o Streaming API e Platform Events para capturar eventos de transação e refletir mudanças no UI.

    Testes de Conhecimento: Como otimizar um LWC para grandes volumes de dados? Quando usar Batch Apex versus Queueable ou Future?

    Exercícios Práticos:

    • Implementar carregamento de dados em páginas (infinite scroll) no LWC.
    • Documentar uma arquitetura event‑driven para processamento de transações.

    Nível Especialista/Consultor

    Explicação: Governança e arquitetura global para fins regulatórios e continuidade operacional.

    Caso de uso: Um banco global precisa garantir data residency e continuidade em cinco países, atendendo a regulações locais.

    Tópicos teóricos:

    • Multi‑org vs Multi‑divisão: trade‑offs, gerenciamento de usuários e compliance em diferentes regiões.
    • Data Residency & Compliance: segmentação de dados por região e criptografia em repouso.
    • Disaster Recovery & Business Continuity: estratégias de backup, restore e failover.

    Hands‑on Lab:

    • Projetar uma solução multi‑org para cinco jurisdições com requisitos de data residency distintos.
    • Configurar um backup externo via APIs do Salesforce e validar o procedimento de restore.
    • Criar documentação detalhada de BCP (Business Continuity Plan) incluindo etapas de failover.

    Testes de Conhecimento: Quais considerações de GDPR impactam a arquitetura multi‑org? Como automatizar backups e restore utilizando APIs?

    Exercícios Práticos:

    • Desenvolver um script que extraia metadados e dados críticos periodicamente.
    • Documentar um plano de BCP com RTO (Recovery Time Objective) e RPO (Recovery Point Objective) definidos.
