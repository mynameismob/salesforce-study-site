    Fundamentos da Plataforma e Integrações

    Esta seção aborda os fundamentos da plataforma Salesforce — arquitetura, dados, automação e desenvolvimento — e também apresenta padrões e ferramentas de integração. Esses conceitos são a base para trabalhar com qualquer nuvem Salesforce.

    Nível Iniciante

    Explicação: A plataforma Salesforce opera em um modelo multi‑tenant, onde metadados definem o comportamento da aplicação e os dados são armazenados em objetos. A segurança é gerenciada em vários níveis para garantir governança de acesso.

    Caso de uso: Uma consultoria precisa rastrear projetos de clientes, incluindo datas de início, término e status.

    Tópicos teóricos:

    • Arquitetura CRM: modelo multi‑tenant e personalização via metadados.
    • Objetos padrão vs custom: quando criar um objeto personalizado vs um campo personalizado.
    • Segurança de dados: Profiles, Permission Sets, Roles e Sharing Rules.
    • Navegação: Lightning Experience, App Launcher e Setup Quick Find.

    Hands‑on Lab:

    • Criar um objeto custom Project com campos StartDate (Date), EndDate (Date), Status (Picklist: Planning, In Progress, Completed) e ClientName (Text).
    • Configurar o layout de página e campos obrigatórios.
    • Criar um perfil Project Manager e conceder permissão de leitura/gravação em Project.
    • Criar um usuário de teste nesse perfil e validar o acesso.

    Testes de Conhecimento: Liste três diferenças entre Profile e Permission Set. Explique o fluxo de avaliação de Sharing Rules.

    Exercícios Práticos:

    • Criar uma Permission Set Read Only Project que permita apenas visualização de registros de Project.
    • Ajustar o layout de página para agrupar campos em seções “Informações Básicas” e “Status”.

    Nível Intermediário

    Explicação: Automação declarativa e modelagem adequada de dados reduzem a necessidade de código e permitem adaptação rápida a requisitos de negócio.

    Caso de uso: Atualizar automaticamente o status de um projeto para Completed sete dias após o término planejado.

    Tópicos teóricos:

    • Modelagem de dados: Master‑Detail versus Lookup e impactos em compartilhamento.
    • Flow Builder: tipos Autolaunched, Record‑Triggered e Screen Flows.
    • Relatórios & Dashboards: tipos tabular, summary, matrix e joined.
    • Importação e exportação de dados: Data Loader, Import Wizard e Data Import Wizard.

    Hands‑on Lab:

    • Criar um Flow record‑triggered em Project que verifica: EndDate + 7 dias < Hoje AND Status != Completed → Atualizar Status = Completed.
    • Criar um relatório de projetos “In Progress” agrupado por ClientName.
    • Agendar um dashboard semanal para exibir a contagem de projetos por status.

    Testes de Conhecimento: Quando usar Master‑Detail em vez de Lookup? Qual tipo de relatório usar para comparar dados de dois objetos?

    Exercícios Práticos:

    • Modificar o Flow para enviar um alerta por email ao proprietário antes da atualização.
    • Importar 50 registros de Project via Data Loader a partir de um arquivo CSV.

    Nível Avançado

    Explicação: Apex e APIs permitem integrações complexas e lógica avançada, essenciais para atender requisitos específicos de negócio.

    Caso de uso: Consumir uma API externa que retorna o status de implementação de um projeto e gravar no campo ExternalStatus__c de Project.

    Tópicos teóricos:

    • Apex Triggers & Classes: estrutura, variáveis de contexto e bulkificação.
    • Testes unitários: uso de @isTest, Test.startTest() e exigência de cobertura mínima de 75%.
    • SOQL & SOSL avançados: consultas com agregações e subqueries.
    • APIs REST/SOAP: autenticação OAuth, Named Credentials e callouts HTTP.
    • DevOps: Salesforce DX, CLI e versionamento de metadata com Git.

    Hands‑on Lab:

    • Escrever um Trigger after insert em Project que cria uma Task “Follow Up” com due date = StartDate + 7 dias.
    • Implementar uma classe Apex para callout REST a um endpoint fictício, atualizando o campo ExternalStatus__c.
    • Criar testes unitários cobrindo cenários de sucesso e falha.

    Testes de Conhecimento: O que é bulkification e por que é essencial em triggers? Qual a diferença entre callouts síncronos e assíncronos?

    Exercícios Práticos:

    • Criar um Batch Apex para atualizar o status de projetos vencidos.
    • Configurar uma Named Credential para autenticação OAuth em um serviço externo.

    Padrões de Integração e Ferramentas

    Além das técnicas de desenvolvimento, integrações multi‑cloud empregam diversos padrões e ferramentas:

    • Processo Batch: Exportar/importar dados em volume via Data Loader ou integradores como MuleSoft.
    • Requisição‑Resposta: Callouts síncronos para serviços externos.
    • Event Driven: Platform Events e Change Data Capture para transmitir alterações em tempo real.
    • Remote Process Invocation: Uso de External Services e Flow Builder para orquestrar APIs externas sem código.
    • MuleSoft e Salesforce Connect: Integrações via middleware e objetos externos.
    • Heroku & Data Cloud: Construção de microsserviços e unificação de dados de múltiplas fontes.

    Boas práticas incluem escrever código Apex bulkificado, usar Named Credentials para centralizar autenticações, planejar deploys com pipelines CI/CD e documentar endpoints e fluxos de dados.

    Nível Especialista/Consultor

    Explicação: Projetar soluções escaláveis exige governança, estratégias de CI/CD e cuidado com segurança e compliance.

    Caso de uso: Implantação global de uma org única para uma empresa com 10 divisões, cada uma com regras de acesso específicas.

    Tópicos teóricos:

    • Padrões Multi‑org: integração federada, arquitetura de namespaces e isolamento de dados.
    • Governança & DevOps: comparação de Change Sets com pipelines CI/CD usando Salesforce DX e SFDX.
    • Segurança & Compliance: Shield Platform Encryption, Event Monitoring e Field Audit Trail.
    • Estrategias de adoção: treinamento de usuários e gestão da mudança.

    Hands‑on Lab / Estudo de Caso:

    • Definir um modelo de dados para suportar divisões de negócio isoladas dentro da mesma org.
    • Configurar um pipeline CI/CD simples usando SFDX para mover metadata de Dev → QA → Produção.
    • Utilizar Shield para criptografar o campo ClientSSN__c e validar o acesso por usuários sem permissão.

    Testes de Conhecimento: Quando usar Change Sets em vez de CI/CD? Quais tipos de dados podem ser criptografados pelo Platform Encryption?

    Exercícios Práticos:

    • Documentar o processo de release management em cinco sandboxes.
    • Configurar monitoramento de login com Event Monitoring para detecção de anomalias.
